# 零基础AI项目教练 · 重新设计方案（完整版）

> 本文档基于原始产品说明书（十八章），结合推广文效果分析、初版代码诊断、以及志远/AI职业风险两个成功项目的经验，重新设计整个产品。

---

## 一、为什么推倒重来

### 1.1 推广文的教训

我们写了一篇付费教程文章《零代码基础，如何用 AI 做出一个完整的软件产品？》，效果极差。根本原因：

**文章卖的是"地图"，用户需要的是"导航仪"。**

具体问题：

- **门槛幻觉**：标题说"零基础"，但用户需要安装 Claude Pro、Claude Code、VS Code、Git、Node.js、GitHub……光准备清单就劝退 90% 读者。
- **知识不值钱**：AI 时代，方法论本身没有付费壁垒，用户可以免费问任何大模型。
- **从"读"到"做"的鸿沟太大**：即便读完全文，用户面对的第一个障碍（安装 CLI）就能卡死 80% 的人。文章只能"告诉"，不能"带着做"。
- **目标用户和内容不匹配**：哲学园 50 万粉丝是高知内容消费者，不是想"做软件项目"的人。真正想做的极少数人，也需要的不是知识，而是工具。

### 1.2 初版代码的问题

另一个大模型做的初版，14 个 task 下来，问题严重：

- **技术栈完全不适配**：Next.js + TypeScript + Prisma + PostgreSQL（Supabase）+ Vercel 部署。Vercel 大陆不可用，Supabase 不稳定，Next.js 远超你的维护能力。
- **核心功能没打通**：AI 对话没接入（明确写着"还不会接入真实 AI"），设计书只有占位，任务拆解只有框架。
- **用户体验差**：子页面缺返回键，整体像开发者后台而非面向普通用户的工具，流程不连贯。
- **路线数据是泛泛的占位文案**：安装步骤写的是"按官方安装说明安装"这种废话，零基础用户看了还是不知道怎么做。

### 1.3 核心结论

> 这个产品不该是"又一个教程"或"又一个聊天机器人"，应该是一个**在浏览器里直接带用户走完全流程的导航系统**——从想法到设计书到任务拆解，零安装、零配置、打开就用。

## 二、产品重新定位

### 2.1 产品本质（保留原说明书定义，修正实现路径）

这是一个面向零基础用户的 Web App。它把"环境准备、工具上手、项目定义、设计书生成、开发起步"连成一体。

核心使命不变：
1. 让用户拥有一套能跑起来的 AI 编程组合。
2. 把脑中的软件想法变成可执行的项目设计书，并进入第一轮开发。

**修正的是实现路径**：不用重型全栈架构，用纯前端 + DeepSeek API 实现全部功能。

### 2.2 一句话描述

用户打开网页，输入自己的软件想法，AI 一步步引导他完成项目定义、需求澄清、设计书生成、任务拆解，最终输出一份可以直接交给 AI 编程工具执行的完整方案。已有工具的用户直接做项目；没有工具的用户先走路线选择和安装引导。

### 2.3 核心转变

从"教你怎么做" → "带你做完"。
从卖知识 → 给工具。
从付费文章漏斗 → 免费工具体验漏斗。

### 2.4 对老蝉的商业价值

- 用户先免费体验工具 → 生成设计书 → 想真正开发了 → 引导进付费社群/陪跑服务
- 工具本身就是最好的推广，用户觉得好会自发传播
- 筛选出真正想做项目的人，比卖文章精准得多

## 三、目标用户（保留原说明书）

### 3.1 核心用户

- 不会写代码，但有产品想法的人
- 想把知识、咨询、内容、服务做成工具的人
- 想做副业、小程序、网页工具、选题工具、测评工具、推荐系统的人
- 想在最短时间内做出一个可展示作品集的人

### 3.2 典型用户画像

- 公众号作者，想把内容方法做成工具
- 教育从业者，想做选志愿、择业、测评、课程配套工具
- 普通上班族，想做一个效率小工具
- 小创业者，想先自己验证想法而不是直接找外包

### 3.3 用户共同特点

- 动机很强
- 对 AI 编程有兴趣
- 不想先学完整编程体系
- 容易被环境安装和工具选择卡住
- 需要一个傻瓜式流程，而不是知识百科

## 四、技术栈（全部替换）

| 层 | 原方案 | 新方案 | 理由 |
|---|---|---|---|
| 前端框架 | Next.js + TypeScript | **纯 HTML + CSS + JS** | 你能维护，志远验证过 |
| UI 库 | shadcn/ui + Tailwind | **原生 CSS（暗色主题）** | 同志远风格，无依赖 |
| 后端 | Next.js API Routes | **无后端** | 前端直调 DeepSeek API |
| 数据库 | PostgreSQL + Prisma | **localStorage** | 无需服务器，零成本 |
| 鉴权 | Clerk / Auth.js | **无登录**（MVP） | 降低门槛 |
| AI 接入 | 未接入 | **DeepSeek API** | 大陆可用，便宜，OpenAI 兼容 |
| 部署 | Vercel | **Cloudflare Pages** | 大陆可访问，你已有经验 |
| 域名 | 无 | **coach.zhexueyuan.com** | 复用现有域名 |

### API Key 策略

用户自填 DeepSeek API Key（同志远模式），零服务器成本。页面上提供获取 Key 的步骤说明和链接。

### 数据持久化策略

- 所有项目数据存 localStorage
- 提供「导出项目 JSON」功能，用户可备份
- 提供「导入项目 JSON」功能，用户可恢复
- P2 阶段再考虑云端保存

## 五、产品总体结构（四层漏斗）

保留原说明书的四层结构，但**调整 MVP 优先级**：

### 第一层：组合路线选择（P1，后补）

用户选一套最适合自己的"模型 + 编程工具"路线。

### 第二层：安装与最小命令训练（P1，后补）

系统手把手带用户完成安装、接入、第一条命令。

### 第三层：项目想法结构化（P0，核心）

用户描述想做的软件，系统像项目经理一样多轮提问、收束需求。

### 第四层：设计书与执行路径生成（P0，核心）

系统输出设计书、任务拆解、下一步动作建议。

> **MVP 策略：先做强第三层和第四层**（这是你用志远验证过的独特方法论），第一层和第二层后补。首页提供分流入口，已有工具的用户直接做项目，没有工具的先去选路线。

## 六、主流程设计（九个页面）

### 6.1 入口首页

**目标**：5 秒内讲清产品价值并引导开始。

**核心文案方向**：

> "你有一个软件想法？不需要会写代码，不需要装任何工具。告诉我你的想法，我带你一步步变成可执行的项目方案。"

**页面模块**：
- 顶部导航栏（Logo + 工作台入口）
- Hero 区：标题 + 一句话描述 + 主 CTA
- 两个主按钮：
  - 「开始做项目」→ 想法输入页
  - 「我还没装工具，先选路线」→ 路线选择页
- 三个价值卡片：把想法变清晰 / 生成专业设计书 / 拆解成可执行任务
- 成果预览区：展示一个设计书的示例（截图或渲染）
- "这个工具是怎么工作的"：3 步流程说明
- FAQ（3-5 个常见问题）
- Footer

### 6.2 路线选择页（P1 实现，先做占位）

**目标**：帮助用户选定一条"模型 + 编程工具"的路线。

**页面模块**：
- 返回首页 + 面包屑导航
- 顶部说明："先选一条适合自己的路线，系统会带你完成安装"
- 静态筛选区（按标签过滤：零基础优先 / 国内优先 / 能力优先）
- 路线卡列表（7 条路线，详见第七章）
- 对比抽屉（点击"对比"可展开两条路线横向比较）
- 底部 CTA："我已装好工具，直接做项目"

### 6.3 安装与接入向导页（P1 实现）

**目标**：一步步带用户完成安装与模型接入。

**页面模块**：
- 步骤导航条（第 1 步 / 共 6 步）
- 当前步骤的安装说明（大字、截图、复制按钮）
- 每步只讲一小步要做什么
- 成功检查提示（"如果你看到了 XXX，说明这步成功了"）
- 故障排查折叠区
- 上一步 / 下一步按钮

**六个步骤**（对应原说明书第八章）：
1. 确认设备与系统环境
2. 确认用户是否已有模型账号
3. 安装对应的编程工具
4. 接入模型
5. 完成初始化
6. 进入最小命令练习

### 6.4 最小命令训练页（P1 实现）

**目标**：建立第一次成功体验。

**页面模块**：
- 步骤练习区（展示要输入的命令，一键复制）
- 结果确认区（"你看到了什么？"选择题）
- 成功庆祝（"恭喜！你的 AI 编程环境已经就绪"）
- 错误反馈（"如果报错了，把错误信息贴到下面"）
- CTA："环境就绪，开始做项目！"

**完成标准**（对应原说明书第九章）：
- 成功启动编程工具
- 成功向模型发出第一条任务
- 成功生成一个最简单页面或文件
- 成功在本地看到结果

### 6.5 项目想法输入页

**目标**：零压力地承接用户自然语言想法。

**页面模块**：
- 大文本输入框
  - placeholder："比如：我想做一个帮高考生选专业的 AI 工具"
  - 支持多行
- 辅助提示文字："不需要专业术语，用大白话说就行。你只需要说清楚三件事：想做什么、给谁用、解决什么问题"
- 示例想法卡片（3-4 个，点击自动填入）：
  - "我想做一个帮高考生选专业的 AI 工具"
  - "我想做一个餐厅点餐小程序"
  - "我想做一个帮我整理读书笔记的网页工具"
  - "我想做一个心理测评小程序"
- DeepSeek API Key 输入区（折叠式面板）
  - 简单说明："需要 DeepSeek API Key 来驱动 AI 对话"
  - 获取步骤 + 链接
  - 输入框 + 保存按钮
  - 安全说明："Key 仅存在你的浏览器中，不会上传到任何服务器"
- 按钮：「下一步：让 AI 帮我理清思路」

### 6.6 AI 需求澄清页（核心引擎）

**目标**：通过多轮对话，把模糊想法收束为结构化项目定义。

**这是整个产品最核心的页面。**

**页面布局**：
- 左侧 / 上方（移动端）：聊天对话区
  - 用户消息 + AI 消息，标准聊天 UI
  - 输入框 + 发送按钮
  - AI 每轮只问 1-2 个问题，不信息轰炸
- 右侧 / 下方（移动端）：项目摘要卡
  - 实时更新的结构化字段表
  - 每个字段显示状态：已填 ✅ / 待澄清 ⏳ / 未涉及 ○
  - 用户可直接点击字段编辑
- 顶部：进度指示（"已澄清 8/15 个关键项"）
- 底部 CTA：「项目已清晰，生成设计书」（当关键字段 ≥ 80% 填充时高亮）

**AI 角色设计**：

系统不是自由聊天，而是扮演"项目澄清官"。每一轮都围绕一个目标：帮助用户从模糊想法走向清晰项目定义。

**System Prompt 核心逻辑**：

```
你是一个专业的项目澄清官。用户告诉了你一个软件想法，你的任务是通过提问帮他把想法变成清晰的项目定义。

规则：
1. 每轮只问 1-2 个问题，不要一次问太多
2. 用大白话，不用技术术语
3. 如果用户的回答模糊，追问具体细节
4. 不要替用户做决定，而是给出选项让用户选
5. 每轮对话结束后，更新项目状态 JSON

你需要逐步收集以下字段：
- project_name: 项目名称
- core_goal: 核心目标（这个软件解决什么问题）
- target_users: 目标用户（给谁用）
- use_cases: 使用场景（在什么情况下用）
- input_data: 输入内容（用户提供什么）
- output_result: 输出结果（系统返回什么）
- core_features: 核心功能（第一版必须有的）
- non_core_features: 非核心功能（可以以后做的）
- platform_type: 平台形态（网页 / 小程序 / APP）
- need_login: 是否需要登录
- need_payment: 是否需要支付
- need_admin: 是否需要后台管理
- data_source: 数据来源
- mvp_scope: MVP 范围（第一版做到什么程度）
- tech_difficulty: 技术复杂度（你来评估）

当所有关键字段都收集到后，告诉用户："你的项目已经定义清楚了，可以生成设计书了。"

每轮回复的最后，输出一个 JSON 块，格式如下：
```json
{
  "project_state": {
    "project_name": "...",
    "core_goal": "...",
    ...已收集到的字段...
  },
  "completion": 0.6,
  "next_focus": "use_cases"
}
```
```

**前端解析逻辑**：
- 每次收到 AI 回复，解析末尾的 JSON 块
- 更新右侧摘要卡的字段显示
- 更新顶部进度条
- 当 completion ≥ 0.8 时，底部"生成设计书"按钮高亮

**对话历史管理**：
- 全部存 localStorage，按 project_id 分组
- 页面刷新后恢复对话历史
- 支持"清空对话重新开始"

### 6.7 设计书生成页

**目标**：一键生成完整设计书，用户可查看、修改、导出。

**三个版本**（对应原说明书第十一章）：

**通俗版**：给非技术人看的，大白话，适合发给合伙人或朋友看。

**标准产品版**：规范的 PRD 格式，包含：
- 项目概述
- 项目目标
- 目标用户
- 用户痛点
- 核心场景
- MVP 定义
- 功能模块说明
- 页面与流程说明
- 数据与输入输出说明
- 技术实现建议
- 风险与边界
- 开发阶段建议

**AI 编程版**：专门优化给 Claude Code / Cursor / TRAE 等 AI 编程工具读的版本。格式更结构化，直接可作为 AGENTS.md 或项目说明文件。

**页面模块**：
- 版本切换 Tab（通俗版 / 标准版 / AI编程版）
- Markdown 渲染的设计书正文
- 操作按钮栏：
  - 「复制全文」
  - 「导出 Markdown 文件」
  - 「我要修改」→ 回到澄清页继续对话
  - 「AI 帮我重写某章节」→ 选择章节 + 修改要求
- 底部 CTA：「下一步：生成任务拆解」

**生成逻辑**：
- 调用 DeepSeek API，把项目摘要卡的完整 JSON 作为输入
- 每个版本使用不同的 System Prompt
- 生成后存入 localStorage

### 6.8 执行拆解页

**目标**：把设计书变成任务清单与可执行的 prompt。

**页面模块**：
- 按阶段分组的任务列表：
  - 阶段一：项目骨架搭建
  - 阶段二：核心功能开发
  - 阶段三：打磨优化
  - 阶段四：部署上线
- 每个任务卡包含：
  - 任务编号（task01, task02...）
  - 任务标题
  - 目标（一句话）
  - 具体步骤（列表）
  - 完成标志
- **首轮开发 prompt**：一段可直接复制给 AI 编程工具的完整 prompt，包含项目背景、当前任务、技术约束
- 开发前检查清单：
  - [ ] 已安装编程工具
  - [ ] 已创建项目文件夹
  - [ ] 已初始化 Git 仓库
  - [ ] 已准备好 API Key
- 操作按钮：
  - 「复制首轮 prompt」
  - 「导出全部任务 Markdown」
  - 「返回修改设计书」

### 6.9 工作台页

**目标**：管理多个项目，随时回到任何阶段继续。

**页面模块**：
- 项目列表（卡片式）
  - 项目名称
  - 当前阶段（想法输入 / 需求澄清 / 设计书 / 任务拆解）
  - 完成度百分比
  - 最后更新时间
  - 「继续」按钮
  - 「删除」按钮（需确认）
- 「创建新项目」按钮
- 数据管理：
  - 「导出所有项目数据」（JSON）
  - 「导入项目数据」（JSON）

## 七、组合路线卡设计（保留原说明书七条路线）

### 7.1 设计原则

路线卡统一呈现"模型 + 编程工具 + 起步方式"的完整组合。优先推荐"模型 + 官方编程工具"组合。

### 7.2 统一字段

每张路线卡都包含：
- 组合名称
- 一句话说明
- 适合谁
- 使用条件
- 安装难度（简单 / 中等 / 中高）
- 你将完成什么
- 推荐标签
- 「选择这条路线」按钮

### 7.3 七条路线（按起步难度排序）

| 排序 | 路线 | 标签 | 安装难度 |
|---|---|---|---|
| 1 | DeepSeek + OpenCode | 新手优先 | 简单到中等 |
| 2 | 通义千问 + 通义灵码 | 官方路线推荐 | 简单到中等 |
| 3 | 豆包/火山方舟 + TRAE | 平台生态路线 | 中等 |
| 4 | 百度千帆/文心 + 文心快码 | 官方路线推荐 | 中等 |
| 5 | ChatGPT + Codex CLI | 进阶推荐 | 中等 |
| 6 | Claude + Claude Code | 工程体验强 | 中等偏高 |
| 7 | Gemini + Gemini CLI | CLI 路线 | 中等 |

> 注意：P1 阶段实现时，需要核实每条路线中工具的最新状态（如 OpenCode 是否仍可用、Codex CLI 的最新安装方式等）。

## 十三、与推广策略的关系

### 做完产品后的推广策略

1. **文章免费，引流到工具**——不再卖付费教程，文章就是推广入口
2. **文章标题方向**：「我 61 岁，做了一个 AI 帮你做软件的工具，你来试试」
3. **用户路径**：读文章 → 点链接体验工具 → 生成设计书 → 想真正开发 → 进付费群/陪跑
4. **工具本身就是推广**：用户体验好了会自发分享

### 付费转化点

- 免费：完整使用工具（从想法到设计书到任务拆解）
- 付费：社群 + 陪跑（带你用设计书做出真正的产品）
- 更高级付费：1对1项目辅导
